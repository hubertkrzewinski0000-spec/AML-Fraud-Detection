# AML Fraud Detection

Financial fraud detection project using **exploratory data analysis**, **SQL-based rule engineering**, and **machine learning** on a 6.3M transaction dataset.

---

## Overview

This project analyzes the [PaySim dataset](https://www.kaggle.com/datasets/ealaxi/paysim1) ‚Äî a synthetic simulation of mobile money transactions ‚Äî to identify behavioral patterns that distinguish fraudulent transactions from legitimate ones.

The analysis demonstrates a realistic AML (Anti-Money Laundering) workflow: starting from raw data exploration, through hypothesis testing with SQL, to building a classification model that can flag suspicious transactions.

---

## Dataset

| Property | Value |
|---|---|
| Source | PaySim (Kaggle) |
| Rows | 6,362,620 transactions |
| Columns | 11 |
| Fraud rate | ~0.13% (8,213 fraud cases) |
| Time span | 30 days (simulated) |

**Key columns:** `step`, `type`, `amount`, `nameOrig`, `oldbalanceOrg`, `newbalanceOrig`, `nameDest`, `oldbalanceDest`, `newbalanceDest`, `isFraud`, `isFlaggedFraud`

---

## Key Findings

- **Fraud is type-specific** ‚Äî 100% of fraud occurs only in `TRANSFER` and `CASH_OUT` transactions. Other types (`PAYMENT`, `DEBIT`, `CASH_IN`) contain zero fraud cases.
- **Balance draining pattern** ‚Äî Fraudulent senders consistently deplete their entire account balance. Average post-transaction balance drops from ~$1.65M to ~$192K, versus legitimate senders who maintain steady balances.
- **High value ‚âÝ Fraud** ‚Äî The top 10 largest transactions (all over $60M) are legitimate. Simple amount-based rules produce unacceptably high false-positive rates.
- **No repeat offenders** ‚Äî Each fraudulent account appears exactly once, meaning structuring (splitting transactions to avoid detection) is not present in this dataset.

## Model Results

The Random Forest model is trained exclusively on `TRANSFER` and `CASH_OUT` transactions with engineered features:

- `balance_drained` ‚Äî whether the sender's balance reached zero
- `dest_balance_no_change` ‚Äî destination balance unchanged despite receiving funds (money laundering signal)
- `amount_to_orig_ratio` ‚Äî transaction size relative to sender's balance

Class imbalance is addressed with `class_weight='balanced'`. Model is evaluated using **Precision, Recall, F1-score** and **ROC-AUC** ‚Äî accuracy alone is misleading on a 99.87% / 0.13% split.
